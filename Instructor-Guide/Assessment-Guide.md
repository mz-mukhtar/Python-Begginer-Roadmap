# 📝 Student Assessment Guide

Assessment in the **Python Beginner to Practical Developer Roadmap** is designed to evaluate practical developer competencies, problem-solving skills, conceptual understanding, and consistent growth—not memorization.

---

## Assessment Components

We recommend distributing course grading across practical coding milestones rather than written exams:

| Assessment Area | Weight | Description |
| --- | :---: | --- |
| **Exercises** | 15% | Completion and effort across weekly guided session exercises |
| **Quizzes** | 10% | Conceptual knowledge checks from weekly session quizzes |
| **Weekly Projects** | 25% | Cumulative iterations of the Student Management System |
| **Participation** | 10% | Engagement, collaboration, questions, and classroom effort |
| **Final Project** | 30% | Architecture, code quality, and functionality of the capstone project |
| **Final Presentation** | 10% | Clear communication and live demonstration of the capstone |
| **Total** | **100%** | |

> **Note**: Instructors may adjust these percentage weights slightly to fit institutional or program grading policies.

---

## Exercise Assessment

Evaluate weekly guided exercises based on consistent effort and practical progress:
- **Attempt Completed**: Student attempted all structured exercises rather than leaving files blank.
- **Logic**: Program workflow follows clear algorithmic steps.
- **Correctness**: Code executes cleanly and generates the expected output.
- **Readability**: Code uses descriptive naming and consistent 4-space indentation.
- **Improvement**: Student incorporates feedback from previous sessions.

---

## Quiz Assessment

- **Use quizzes as learning checks**: Weekly quizzes (`quiz.md`) should serve as low-stakes self-assessments to reinforce key terminology.
- **Review wrong answers**: Always review commonly missed quiz questions together at the start of the next session.
- **Avoid punishment-focused assessment**: Do not penalize students heavily for early conceptual mistakes.

---

## Weekly Project Assessment

Use this standardized 100-point rubric to evaluate weekly project submissions (`weekly-projects/week-*`):

| Area | Points | Criteria |
| --- | :---: | --- |
| **Program runs** | 20 | Application launches cleanly from terminal without startup syntax errors |
| **Required features** | 30 | All weekly menu options and functional requirements are implemented |
| **Correct concepts** | 20 | Code utilizes the specific week's pedagogical focus (e.g., loops, files, OOP) |
| **Readability** | 10 | Variable/function names are descriptive; code is cleanly formatted |
| **Error handling** | 10 | Where applicable, code handles invalid inputs safely without crashing |
| **Student explanation** | 10 | Student can explain how their project code works in plain spoken words |
| **Total** | **100** | |

---

## Participation Assessment

Assess classroom participation based on holistic effort and collaborative attitude:
- **Attendance**: Consistent presence across sessions.
- **Practice**: Active typing and engagement during practical coding blocks.
- **Questions**: Willingness to ask clarifying questions when stuck.
- **Collaboration**: Supportive peer communication during pair programming.
- **Effort**: Perseverance when troubleshooting complex traceback errors.

> **Important**: Do not reward only vocal or advanced students. Recognize students who show steady dedication and improvement.

---

## Formative Assessment

Formative assessment occurs continuously **during learning** to check comprehension and guide immediate instruction:
- **Questions**: Asking students to explain what a line of code does during live coding.
- **Observation**: Walking around the classroom while students work on starter files.
- **Short exercises**: 10-minute warm-up coding prompts.
- **Code review**: Informal peer or instructor reviews during session practice.
- **Exit tickets**: Asking students to name one concept they mastered and one question they still have before leaving.

---

## Summative Assessment

Summative assessment evaluates student mastery **after a learning stage** has concluded:
- **Weekly project**: Verifying applied mastery of the week's topic.
- **Mid-course practical**: Comprehensive practical check after Week Four.
- **Final project**: Capstone application demonstrating full 8-week synthesis.
- **Presentation**: Live technical showcase and code walkthrough.

---

## Mid-Course Assessment (End of Week Four)

We recommend conducting a practical mid-course checkpoint at the conclusion of Week Four.
- **Requirements**: Students build a simple, file-based Python application from scratch (or extend their Week Four Student Management System).
- **Competencies Assessed**:
  - Variables, data types, and arithmetic/logical operators
  - Conditional branching (`if`/`elif`/`else`)
  - Iteration (`while` and `for` loops)
  - Data collections (`list`, `dict`, `set`)
  - Single-responsibility functions and docstrings
  - Local file persistence (`pathlib`, JSON, CSV)

---

## Final Assessment

The primary final assessment is the capstone software project evaluated using the comprehensive **100-Point Final Project Evaluation Rubric** (`Project-Rubric.md`).

---

## Feedback Guidelines

Effective instructor feedback builds confidence and accelerates learning. Ensure feedback is:
- **Specific**: Point out exact functions or lines of code.
- **Respectful**: Validate student effort before offering constructive critique.
- **Actionable**: Provide clear, step-by-step guidance on how to refactor or fix the issue.
- **Balanced**: Highlight what was done well alongside what needs improvement.
- **Timely**: Deliver feedback promptly while the assignment is fresh in the student's mind.

### Feedback Examples
- **Avoid Vague/Discouraging Critique**:
  ```text
  Your code is bad.
  ```
- **Prefer Actionable, Constructive Feedback**:
  ```text
  Your search feature works well.
  Try moving the repeated input code into a function to improve organization.
  ```

## Student Progress Status

Track individual student development using four standardized progress levels:

```text
Needs Support      Student struggles with core syntax and needs scaffolded 1-on-1 assistance
Developing         Student understands basic syntax but needs guidance on architecture and debugging
Meeting Expectations Student independently writes, debugs, and delivers required features cleanly
Exceeding Expectations Student implements robust error handling, tests, and clean architecture autonomously
```

## Academic Integrity & Responsible AI Guidance

Maintaining academic integrity ensures students develop authentic programming abilities.

### Collaboration Policies
- **Students May**:
  - Discuss high-level application ideas and data structures with peers
  - Ask instructors or peers clarifying questions about syntax or tracebacks
  - Consult official Python documentation and course cheat sheets
- **Students Should Not**:
  - Copy complete project files from peers or external repositories
  - Present another person's code as their own original work
  - Submit generated code without understanding how it works

### Responsible AI Assistance
Students may use AI tools as interactive learning tutors when used responsibly:
- **Encouraged AI Usage**:
  - Ask for line-by-line explanations of confusing syntax
  - Ask for help understanding what a traceback error message means
  - Ask for small conceptual examples of specific methods
  - Compare alternative approaches to solving a logic problem
- **Discouraged AI Usage**:
  - Copying complete capstone projects directly from an AI prompt
  - Submitting AI-generated code without testing or reading it
  - Sharing private personal credentials or API keys with AI models

> **Core Requirement**: A student should be able to explain every important part of submitted code in their own words during code review.
